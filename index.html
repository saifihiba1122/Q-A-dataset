The constraint

Big book, tiny budget.

A free Gemini key allows only a handful of requests per day. A naïve script either burns through the quota mid-run and loses everything, or silently exceeds it. The whole design is shaped by that single limit.

Challenge

Free-tier daily cap

Every request counts — including failed retries, which the API still bills. A counter that only tracks successes drifts below real usage and blows the cap without warning.

Challenge

Half a million words

The book is far too large for one prompt, and small chunks would need hundreds of calls. Chunk sizing had to fit the entire job inside the daily budget.

Challenge

Runs span days

One day's quota isn't enough, so the job must survive being stopped and resumed — no duplicate work, no lost records.

Challenge

Messy model output

LLMs wrap JSON in markdown, truncate long replies, and repeat themselves. Raw output can't be trusted as a dataset.

The pipeline

Six stages, one command.

Each Markdown file flows through two passes — a precise pass for Q&A and a broad pass for summaries — before becoming validated JSONL ready for the Hub.

Why it's robust

The engineering, not just the prompt.

The interesting work isn't asking an LLM for questions — it's making the job safe, repeatable, and impossible to run over budget.

01

Quota-honest counting

Every call is counted before it's made — successes and retries alike — so the internal counter can never sit below real Google-side usage. A hard ceiling stops the run at the budget, period.

02

Crash-safe resume

Progress is written to disk after every chunk with idempotent task IDs. Stop it, crash it, close the terminal — the next run skips finished work and continues exactly where it left off.

03

Dual chunking

Two passes over each document: small chunks give Q&A precision, large chunks give summaries the full context of a section. One size never fits both jobs.

04

Trustworthy output

JSON mode plus a high token ceiling stop mid-reply truncation; validation drops empty or trivial answers; instruction hashes de-duplicate records across runs.

05

Config-driven

Paths, chunk sizes, model, rate limits and Hub settings all live in config.yaml. Day-to-day tuning never touches the Python.

06

Secrets stay out

The API key lives in a git-ignored .env; only the code and docs reach GitHub, the dataset goes to the Hub. Nothing sensitive is ever committed.
